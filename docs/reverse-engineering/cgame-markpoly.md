# cgame `markPoly_t` Layout Map

This note maps the retail-compatible x86 layout of `markPoly_t` used by
`cgamex86.dll` onto the current `src/code/cgame/cg_local.h` definition. The
goal is to separate hard layout facts from the higher-level roles played by the
mark pool, active/free ownership lists, and the persistent fade/render path in
`cg_marks.c`.

## Method

- Layout facts come from a local x86 record-layout dump of `markPoly_t`,
  `poly_t`, and `polyVert_t` using
  `clang -target i686-pc-windows-msvc -DID_INLINE=__inline -Xclang -fdump-record-layouts`
  against `src/code/cgame/cg_local.h` and `src/code/cgame/tr_types.h`.
- Member roles were cross-checked against the live mark lifecycle in
  `src/code/cgame/cg_marks.c`: `CG_InitMarkPolys`, `CG_FreeMarkPoly`,
  `CG_AllocMark`, `CG_ImpactMark`, and `CG_AddMarks`.
- This pass maps the persistent mark payload only. Temporary marks generated by
  `CG_ImpactMark` bypass `markPoly_t` entirely and are submitted directly to the
  renderer.

## Hard Layout Facts

- `sizeof(polyVert_t) = 0x18` (`24`).
- `sizeof(poly_t) = 0x0C` (`12`).
- `sizeof(markPoly_t) = 0x120` (`288`).
- `MAX_VERTS_ON_POLY = 10`, so the trailing vertex slab is
  `10 * sizeof(polyVert_t) = 0x0F0`.
- The high-level banding is:
  - list ownership links at `0x000`
  - lifetime/shader/fade metadata at `0x008`
  - persistent fade baseline color at `0x014`
  - embedded `poly_t` header at `0x024`
  - inline `polyVert_t[10]` storage at `0x030`

## Top-Level Member Map

| Offset | Member | Type | Role |
| --- | --- | --- | --- |
| `0x000` | `prevMark` | `markPoly_t *` | Previous node in the active doubly linked list; unset for free-list nodes. |
| `0x004` | `nextMark` | `markPoly_t *` | Next node in the active doubly linked list, and the only link used by the free list. |
| `0x008` | `time` | `int` | Allocation timestamp used for oldest-mark eviction and fade/expiry timing. |
| `0x00C` | `markShader` | `qhandle_t` | Persistent renderer shader handle passed to `trap_R_AddPolyToScene`. |
| `0x010` | `alphaFade` | `qboolean` | Chooses alpha-only fade versus RGB fade during the final `MARK_FADE_TIME`. |
| `0x014` | `color[4]` | `float[4]` | Persistent normalized RGBA baseline used to recompute per-vertex fade colors. |
| `0x024` | `poly` | `poly_t` | Embedded compatibility header; in the persistent mark path only `poly.numVerts` is strongly live. |
| `0x030` | `verts[10]` | `polyVert_t[10]` | Inline persistent copy of the clipped decal polygon submitted every frame until expiry. |

## Embedded `poly_t` And `polyVert_t`

### `poly_t` at `mark->poly`

| Offset in `poly` | Member | Type | Role |
| --- | --- | --- | --- |
| `0x000` | `hShader` | `qhandle_t` | Structurally present, but no strong live owner was revalidated in the persistent `markPoly_t` path. |
| `0x004` | `numVerts` | `int` | Live vertex count copied from the clipped mark fragment and reused by fade/render loops. |
| `0x008` | `verts` | `polyVert_t *` | Structurally present, but no strong live owner was revalidated in the persistent `markPoly_t` path. |

### `polyVert_t` at `mark->verts[i]`

| Offset in `polyVert_t` | Member | Type | Role |
| --- | --- | --- | --- |
| `0x00` | `xyz` | `vec3_t` | Clipped mark vertex position in world space. |
| `0x0C` | `st[2]` | `float[2]` | Projected texture coordinates derived from the decal axes and radius. |
| `0x14` | `modulate[4]` | `byte[4]` | Per-vertex RGBA written at impact time and then mutated by the fade logic. |

## Ownership Notes

### Pool And List Ownership

- `cg_markPolys[MAX_MARK_POLYS]` is the fixed backing pool for all persistent
  marks.
- `cg_activeMarkPolys` is a sentinel head for a doubly linked active list. Its
  `nextMark` and `prevMark` fields self-point after `CG_InitMarkPolys`.
- `cg_freeMarkPolys` is a singly linked free list that only consumes
  `nextMark`.
- `CG_AllocMark` always succeeds by freeing the oldest active marks when the
  free list is empty. The oldest-age comparison is keyed off `time`.
- `CG_FreeMarkPoly` removes a live node from the active list and pushes it onto
  the free list without preserving any render payload.

### Persistent Mark Payload

- `CG_ImpactMark` only allocates a `markPoly_t` for non-temporary marks.
- `time`, `alphaFade`, `markShader`, `poly.numVerts`, `color[4]`, and `verts[]`
  are the fields populated for persistent marks.
- `verts[]` receives a full inline copy of the clipped fragment vertices via
  `memcpy`, which is why `markPoly_t` carries its own `polyVert_t[10]` slab
  instead of relying on external storage.
- `color[4]` stores normalized float channels, while `verts[].modulate[]`
  stores byte channels. Retail keeps both because the fade logic repeatedly
  recomputes byte colors from the float baseline.

### Fade And Render Behavior

- `CG_AddMarks` expires a mark once `cg.time > time + MARK_TOTAL_TIME`.
- The special energy-mark path compares `markShader` against
  `cgs.media.energyMarkShader` and fades the RGB channels early while leaving
  alpha behavior to the later general fade block.
- The general fade block uses `alphaFade` to choose between:
  - reducing `modulate[3]` only, or
  - recomputing `modulate[0..2]` from `color[0..2]`
- Every surviving mark is rendered through
  `trap_R_AddPolyToScene( markShader, poly.numVerts, verts )`.

### Embedded `poly_t` Compatibility Header

- The persistent mark path strongly uses `poly.numVerts`, but the current tree
  does not show matching live writes or reads for `poly.hShader` or `poly.verts`.
- In practice, `markShader` is the authoritative shader handle and the outer
  `verts[]` slab is the authoritative vertex storage.
- The safest reading is that retail keeps `poly_t` embedded for structural
  compatibility with generic polygon helpers, but the persistent wall-mark path
  only needs the count field from that header.

## Practical Reading Guide

- If the question is "who owns this mark right now?", start at `prevMark` /
  `nextMark`.
- If the question is "when will this mark be reclaimed or fade?", start at
  `time` and `alphaFade`.
- If the question is "what actually gets submitted to the renderer?", start at
  `markShader`, `poly.numVerts`, and `verts[]`.
- If the question is "why are there both float colors and byte colors?", start
  at `color[4]` versus `verts[].modulate[4]`.
